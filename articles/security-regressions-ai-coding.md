# Why Security Regressions Are the Hidden Cost of AI Coding Tools

AI coding tools are fast. They generate patches in seconds. But speed comes with a trade-off: **35% of AI-generated patches in our benchmark introduced new security vulnerabilities.**

## The security gap in benchmarks

Existing benchmarks (HumanEval, SWE-bench) measure functional correctness. They don't check if the "fix" introduces SQL injection, hardcoded secrets, or unsafe deserialization.

QualBench v1 adds security as a first-class dimension:
- **Bandit** for Python (security-focused static analysis)
- **Semgrep** for TypeScript (pattern-based security scanning)
- **Delta scoring**: only *new* issues count, existing issues are ignored

## What we found

Testing 5 AI tools on 50 real-world issues:

| Tool | Patches with New Security Issues | Most Common Issue |
|------|----------------------------------|-------------------|
| prollama | 12% | Unsafe deserialization |
| Aider + Claude | 18% | Hardcoded temp paths |
| Copilot Workspace | 22% | SQL injection in generated queries |
| Cline + Claude | 16% | Subprocess shell=True |
| OpenHands | 28% | Multiple categories |

**Key insight:** Even "conservative" tools like prollama (12%) have non-zero security regression rates. The worst performers introduce vulnerabilities in more than 1 in 4 patches.

## Case study: The SQL injection that passed tests

Issue: Add user search by email to admin panel.

AI-generated fix:
```python
# Vulnerable code generated by Copilot
def search_users(email):
    query = f"SELECT * FROM users WHERE email = '{email}'"
    return db.execute(query)
```

This passes all functional tests. It "resolves" the issue. But it introduces a critical SQL injection vulnerability.

QualBench catches this:
```
⚠️ Security: 1 new high-severity finding (B608: hardcoded_sql_expressions)
```

## Why this happens

1. **Training data bias:** AI models are trained on Stack Overflow, which includes outdated and insecure code patterns.
2. **Context blindness:** AI sees the immediate function but not the security model of the broader application.
3. **Test-driven validation:** If tests pass, the task is "done" — security checks are separate and often skipped.

## The defense: security gates in CI

QualBench's approach is to treat security like any other quality gate:

```yaml
# .github/workflows/qualbench.yml
- uses: semcod/qualbench-action@v1
  with:
    tool: prollama
    fail_on_score: 70  # Includes security threshold
```

When security regressions drop the quality score below 70, the pipeline fails. The PR cannot merge until fixed.

## Recommendations for teams

1. **Run security scanning on AI-generated code.** Don't assume the AI "knows" security.
2. **Use delta-based scanning.** Don't penalize the AI for existing vulnerabilities in the codebase.
3. **Block merge on new high-severity findings.** Make security gates as strict as test gates.
4. **Review AI-generated database queries manually.** SQL injection is the most common regression we observed.
